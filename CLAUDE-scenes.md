### Room zones

Each room (e.g. `HCZ_049`, `LCZ_173`, `EZ_GateA`) carries `RoomIdentifier` + `CullableRoom`. Variants are duplicated as `(1)…(13)` copies (e.g. 14× `HCZ_Room3`, 11× `LCZ_TCross`). Map generation (`SeedSynchronizer` → `ImageGenerator` per zone) repositions/activates these pre-placed rooms — rooms are **scene objects, not runtime-instantiated prefabs**.
- HeavyRooms also contains `SCP106_PORTAL` (Animator, NetworkIdentity) and `PocketWorld` (the pocket dimension room, `RoomIdentifier` + `CullableRoom`).
- Outside contains: `Root_*&*Outside Cams` (43 children, `SafeTeleportPosition`), surface `LCZ/EZ BreakableDoor`s, `RESPAWN EFFECTS CONTROLLER` (`RespawnEffectsController`, NetworkIdentity), `FlickerableLightController` (NetworkIdentity + `NetIdWaypoint`), `CCTV` (12 children), `Chopper`, `Clutter` (`ClutterSpawner`).

### Main Camera Controller — FPS camera stack ⚠ CRITICAL FOR VIEWMODELS

```
Main Camera Controller            (MainCameraController)
└── Default Camera
    └── All Cameras
        └── ViewmodelCamera       (Camera, AudioSource, ChangeSettingsOnOutside,
            │                      ViewmodelCamera script, CenterScreenRaycast)
            └── Item Workspace    (SharedHandsController)
                └── … Hands Rig + viewmodel workspace
```

- **Viewmodels are rendered by a separate `ViewmodelCamera`**, not the main world camera. Anything that must show in first person (weapon viewmodel, Hands Rig arms) must be visible to THIS camera (its cullingMask / layer).
- `SharedHandsController` (singleton driving the Hands Rig two-animator system, see CLAUDE-firearm.md) lives on `Item Workspace`, a child of ViewmodelCamera.
- Several weapon viewmodel prefabs carry `CameraRenderFlag` + their own extra Cameras (scope rendering via `ScopeShaderReplacement`).

## Radial Inventory UI (`RadialInventory` : `RadialMenuBase`)

The main inventory is a **radial menu** (`Inventory Canvas → InventoryCirclePart`). Geometry — verified 1:1 against the original build (`level2`), do **not** "fix" by resizing:

- **`RingImage`** (GO `7836`, RT `67995`): the donut background, `sizeDelta 785×785`, sprite from `RadialMenuSettings.MainRings[slotCount]`.
- **`Slot Template`** (GO `12089`, Image `74017`, RT `69100`): `sizeDelta 785×785` (= full ring), `scale 1`, color alpha 0. **This is the per-icon highlight/background template, NOT a per-slot-sized box.**
  - `RadialMenuBase.GetHighlightSafe()` instantiates it once per slot as `Highlights[i]`, parents to `RingImage`, sets `localPosition = 0` and `localEulerAngles = angleStep·i` (rotation only), and assigns sprite `RadialMenuSettings.HighlightTemplates[slotCount]`.
  - The highlight sprites are `Assets/Sprite/SlotN.asset` — **polygon-mesh wedges** (≈1/N of the ring) with an **off-center pivot/offset** (e.g. Slot5 pivot `(-0.017,-0.227)`, offset `(-486,-600)`) on a `SlotN.png` 2152² sheet. The off-center pivot is what pushes the wedge out to the ring edge; it **only lines up at the full 785×785 rect**. Shrinking `Slot Template` collapses every wedge toward the centre — so a "full-size background" symptom is **never** fixed by resizing the template; suspect the wedge sprite/texture or the per-state tint in `RadialInventory.RefreshItemColors` (`_circleColor/_highlightColor/_heldColor/_blockedColor/_wornColor` → `highlight.color`).
- Facility `RadialInventory` has **8** `_iconSlot`s (`RawImage` item icons) → uses `HighlightTemplates[8]`. `HighlightTemplates[0]`/`[1]` are the full placeholder sprite (`bf082267…`), used only at slot count ≤1.
